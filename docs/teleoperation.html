Teleoperation

Teleop first aligns the follower to the current leader pose with a safety-checked SDK move, then feeds leader targets into the SDK online direct joint stream. Studio keeps the 30 Hz leader, camera, and recording sample contract while soarm-sdk owns fixed-rate follower output, latest-target writes, joint-limit checks, and timeout hold. 

soarm-studio teleop --config configs/session.yaml --free-test --seconds 5

Use --free-test after calibration. It runs readiness checks before starting the loop.

Reading the CLI output

A successful teleop command prints the hardware connection log, then a JSON metrics object. The JSON describes the Studio-side synchronous leader, follower, and camera sampling loop, not the SDK's internal direct stream output rate.

Example 60 second free-test output

Field Meaning
iterations How many synchronized Studio samples were completed. At 30 Hz for 60 seconds, a healthy run would be near 1800.
target_hz The configured Studio loop target from configs/session.yaml.
observed_hz The actual Studio loop rate, computed as iterations / elapsed_s.
last_latency_ms The time spent in the last loop step, including leader read, follower readback, target update, and camera reads if cameras are configured.
max_latency_ms The slowest loop step seen during the run.
elapsed_s The wall-clock duration of the teleop run.
!
What the example means

The run connected to both arms and exited normally, but the Studio loop did not keep up with 30 Hz. The actual rate was 14.752 Hz and the last step took 67.854 ms, which is longer than the 33.3 ms budget for a 30 Hz loop. This usually means the synchronous app loop is I/O-bound. Direct streaming removes SDK-side smoothing lag, but the follower still only receives new leader targets as fast as Studio can read and publish them.
