Dataset recording

Before writing a dataset, run teleop without recording and confirm joint directions, usable ranges, gripper behavior, and loop frequency. Start episode capture only after that check passes.

Validate teleop before recording

Complete Arm calibration and run check first. Clear the follower workspace, place both arms near safe mid-range poses, then run a 30-second free test. --free-test repeats readiness checks before the control loop starts.

CLI 
soarm-studio teleop \
  --config configs/session.yaml \
  --free-test \
  --seconds 30

  1. Once the command starts, hold the leader and slowly move the shoulder, elbow, and wrist joints one at a time, then open and close the gripper. Confirm the follower moves continuously in the same direction without hitting software limits before trying combined motion.

  2. If the follower reverses direction, jumps, approaches a mechanical hard stop, or develops unusual lag, press Ctrl-C immediately and do not record. Recheck role bindings, calibration ranges, and check output first.

  3. A healthy run ends in TELEOP_READY, with observed_hz close to the configured target_hz. Proceed to recording only after this passes.

Run teleop diagnostics when needed

If loop frequency is low or motion lags, start with debug output. Add --follower-readback-every 10 only when periodic post-command follower state is needed. See Teleoperation output for metric definitions.

CLI 
soarm-studio teleop \
  --config configs/session.yaml \
  --free-test \
  --seconds 60 \
  --debug

soarm-studio teleop \
  --config configs/session.yaml \
  --free-test \
  --seconds 60 \
  --debug \
  --follower-readback-every 10

Record the dataset

Recording writes a local LeRobot Dataset v3 dataset plus SOARM Studio sidecars describing the hardware session and sync quality. Start with one short episode to verify task text, images, joint data, and output paths before scaling capture.

CLI 
soarm-studio record \
  --config configs/session.yaml \
  --episodes 1 \
  --warmup 1 \
  --seconds 10 \
  --task "pick object" \
  --overwrite \
  --save-policy manual

With --save-policy manual, each episode waits for confirmation before it starts. During capture, press d to end the current episode and review it, or press a to discard it and immediately record the episode again. At the review prompt, press d to save and continue, or a to discard and retry.

Capture timing

Recording uses the Studio target loop tick as the single dataset time base. The first episode's --warmup and capture segment share one TeleopLoop, so Studio does not run a second sync_start or restart the follower stream after warmup. Warmup automatically estimates the joint read lead time and the camera receive-to-estimated-exposure offset; in normal use, keep --warmup 1 instead of tuning extra offsets by hand. 

camera threads:
  camera.read() @ camera.fps -> latest frame

warmup:
  same TeleopLoop
  collect joint request/receive timing
  collect camera receive intervals
  -> estimate joint_read_lead_ms
  -> estimate camera_receive_to_estimated_exposure_ms

episode start:
  enable episode camera history

Studio loop @ loop_hz:
  target_tick[n] = episode_start + n / loop_hz
  sleep until target_tick[n] - joint_read_lead
  leader_read || follower_before_read
  -> action
  -> SDK direct stream update
  -> append ControlSample(joints, action, target_tick)

after episode:
  camera receive timestamp -> estimated exposure timestamp
  target_tick -> nearest estimated exposure timestamp
  -> write LeRobot frame and quality sidecars

This means teleop debug camera_age:* reports latest-frame preview freshness, while recording quality max_camera_age_ms reports the difference between the matched estimated exposure time and the target loop tick. Standard USB/UVC cameras do not provide hardware exposure timestamps here, so this is a software estimate rather than hardware sync.

Normal recording only saves summary quality metrics. Add --debug to the record command when you need the automatic timing model, joint read timing, raw per-camera receive timestamps, estimated exposure timestamps, and per-sample matching offsets; the sidecar is written to episodes/episode_000000/camera_timing.json. Its timing_model contains joint_read_lead_ms and camera_receive_to_estimated_exposure_ms.
