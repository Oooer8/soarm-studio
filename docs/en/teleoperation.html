Teleoperation

Teleop is a 30 Hz Studio-side sampling loop wrapped around the SDK direct joint stream. Studio reads the leader and the follower's pre-action state, computes a clipped target, and publishes that target to soarm-sdk. The SDK owns fixed-rate follower output, latest-target writes, joint-limit checks, and timeout hold.

See teleop validation before dataset recording for the concrete arm-handling sequence and diagnostic commands. This page focuses on the control path and output metrics.

Technical plan

The default live path is optimized for deterministic target publishing: 

leader_read
  -> follower_before_read
  -> target_compute
  -> SDK direct stream update
  -> latest camera frame fetch
  -> sample callbacks
  -> absolute 30 Hz scheduler sleep
Area Decision
Follower output Use the SDK direct joint stream. Studio only sends the newest target; the SDK handles fixed-rate output and timeout behavior.
Follower readback Do not read follower_after every live frame. The control path needs follower_before for state, clipping, and recording. Readback can be enabled for diagnostics with --follower-readback-every N.
Cameras OpenCV cameras run in background latest-frame threads. The control loop fetches cached frames, so camera USB or decode jitter does not block target publishing.
Scheduler Sleep against an absolute 30 Hz tick with a short guard window, so normal OS sleep overshoot does not accumulate into loop drift.
Debug metrics Debug data is opt-in and reported as phase summaries. The default path keeps the same behavior without storing per-phase samples.

The concrete --debug and --follower-readback-every runs also live in the dataset recording tutorial, so commands are maintained in one operational page.

Reading the CLI output

A successful teleop command prints the hardware connection log, then a JSON metrics object. The JSON describes the Studio-side 30 Hz control and sampling loop, not the SDK's internal direct stream output rate.

Field Meaning
iterations How many Studio control samples were completed. At 30 Hz for 60 seconds, a healthy run should be near 1800.
target_hz The configured Studio loop target from configs/session.yaml.
observed_hz The actual Studio loop rate, computed as iterations / elapsed_s.
last_latency_ms The time spent in the last loop step, including leader read, follower-before read, target compute, stream update, latest-frame camera fetches, and callbacks.
max_latency_ms The slowest loop step seen during the run.
elapsed_s The wall-clock duration of the teleop run.
phase_latency_ms Present only with --debug; reports count, last, average, p50, p95, and max latency for each teleop phase. camera_read:* is cached-frame fetch cost, camera_age:* is frame freshness, and scheduler_lag is timing drift.
i
What the example means

The run completed 1800 control samples in 60 seconds with zero over-budget loop steps. Arm reads and target updates are under a few milliseconds, camera reads are cached latest-frame lookups, and scheduler lag stays below one millisecond at p95. The camera-age phases show frame freshness; values around one camera frame are expected for 30 Hz cameras.
